# Takedown 처리 절차 (메인테이너용)

브랜드 권리자(또는 대리인)가 저작권·상표 등을 이유로 카탈로그의 로고·콘텐츠 삭제를 요청했을 때, 메인테이너가 따르는 내부 처리 절차서다.

> [!IMPORTANT]
> 이 문서는 메인테이너의 **자율적 운영 절차**이며 **법률자문이 아니다.** 법적 효력이 필요한 정식 통지(내용증명 등)나 호스팅 사업자 차원의 강제 조치를 원하는 권리자에게는 GitHub의 DMCA 절차([github.com/contact/dmca](https://github.com/contact/dmca))를 안내한다. 실제 분쟁 단계에서는 한국 저작권 전문가의 자문을 받는다.

신고자(외부)용 요약 정책은 [SECURITY.md](../SECURITY.md#브랜드-자산콘텐츠-삭제-요청-takedown)와 [NOTICE](../NOTICE)에 있다. **이 문서는 권리 침해 신고를 처리하는 절차이며, 행동 강령(Code of Conduct) 위반 신고는 [CODE_OF_CONDUCT.md](../CODE_OF_CONDUCT.md)의 별도 경로를 따른다.**

## 접수 채널

| 채널 | 언제 | 비고 |
|------|------|------|
| **공개 이슈** — `⚖️ 삭제(Takedown) 요청` 템플릿 (`legal/takedown` 라벨) | 로고·콘텐츠의 단순·명백한 삭제 요청 | 처리 투명성↑. 신고자에게 민감정보 기입 금지를 안내함 |
| **비공개** — [Security Advisories](https://github.com/CaesiumY/ko-design-md/security/advisories/new) | 법적 분쟁·대리인 관여·NDA/미공개 자료·개인정보·영업비밀, 또는 제거 사실 자체를 비공개로 하고 싶을 때 | 단일 메인테이너에게 직접 전달 |

공개 이슈로 민감한 사안이 들어오면, 비공개 채널로 옮기도록 요청하고 공개 이슈는 식별정보를 편집·정리한다.

## 처리 7단계

### 1. 접수·분류
- 첫 응답 목표: **영업일 기준 72시간 이내**([SECURITY.md](../SECURITY.md)와 동일 SLA).
- 대상 유형 식별: **로고 / 카탈로그 항목 콘텐츠 / 프리뷰·OG / 복합**.
- 민감·법적 사안이면 즉시 비공개 채널로 전환.

### 2. 진위·권리 확인
- 요청자가 **권리자 본인 또는 정당한 대리인**인지 확인한다.
- 확인 수단: 브랜드 **공식 도메인 이메일**, 브랜드 공식 채널과의 대조, (대리인인 경우) 위임 증빙. **증빙은 공개 이슈가 아니라 비공개 채널로** 받는다.
- 진위가 불확실해도 적대적으로 대하지 않는다. 대부분은 선의의 routine 요청이다.

### 3. 범위·근거 파악
- **무엇을**: 파일 단위로 특정한다(어떤 로고/어떤 slug/어떤 섹션).
- **왜**: 저작권 / 상표권 / 브랜드 가이드라인 위반 / 기타.
- **어디까지**: 전체 삭제 vs 부분 추상화로 충분한지.

### 4. 검토
- 콘텐츠는 [CC BY 4.0](../LICENSE-CONTENT), 로고는 [NOTICE](../NOTICE)의 "식별·참조 목적" 한정이라는 점, 공개 출처 기반이라는 점을 확인한다.
- 다만 **단일 개인 OSS가 권리자와 분쟁을 길게 끌 실익은 거의 없다.** 항변 근거가 있어도 분쟁 가치가 낮으면 **협조적·신속 제거를 우선**한다.
- 동시에 **과잉 삭제는 피한다** — 요청 범위에 맞춰 필요한 만큼만 조치한다.

### 5. 조치
조치 후 반드시 `pnpm build`로 파서·빌드 통과를 확인한다. (DCO 서명 `git commit -s`, 커밋 scope 예: `chore(takedown): …`)

**로고만 제거**
- `public/logos/{slug}.{svg|png|webp|avif}` 삭제.
- `services/{slug}.md`의 프론트매터 `logo:` 줄도 제거한다.
- → OG는 `src/og/load-logo.ts`의 **text-only 폴백**으로 자동 처리되어 빌드가 깨지지 않는다. (`logo:`를 남기고 파일만 지우면 빌드 경고가 뜨므로 프론트매터까지 함께 정리한다.)

**항목 콘텐츠 전체 제거**
- `services/{slug}.md` 삭제.
- `public/preview/{slug}/light.html`·`dark.html` 삭제.
- OG PNG(`public/og/{slug}.png`)는 **`.gitignore`된 빌드 산출물**이라 커밋된 파일이 없다 — 재배포 시 자동 반영된다.
- **필수**: 다른 모든 항목의 프론트매터 `related_services`에서 해당 slug를 제거한다. 파서가 `related_services`의 참조 무결성을 검증하지 않으므로(`src/lib/content-parser.ts`), 남겨두면 사이트에 깨진 관련-항목 링크가 잔존한다.
  ```bash
  # 삭제 대상 slug가 다른 항목에서 참조되는지 확인 (0건이어야 함)
  grep -rn "{slug}" services/*.md
  ```

**부분 추상화(전체 삭제 대신)**
- 문제가 된 도메인 특화 컴포넌트명·카피만 일반화한다. 카탈로그의 취지("겉모습은 빌리되 비즈니스는 베끼지 않는다", [README](../README.md))와 정합한다.

**git 이력 처리**
- 기본은 일반 커밋으로 제거한다(이력에는 남는다).
- 영업비밀·개인정보 등으로 **이력에서도 완전 제거**가 필요한 예외에만 `git filter-repo`(또는 BFG) + force-push를 고려한다. 협업 영향이 크므로 **최후 수단**이며, 권리자와 합의한 경우에 한한다.

### 6. 회신
처리 결과를 신고자에게 회신한다. 아래 템플릿을 사안에 맞게 사용한다.

```
[접수 확인]
안녕하세요. 요청을 정상적으로 접수했습니다.
대상: {로고/항목} / 근거: {저작권·상표 등}
권리 확인을 위해 {공식 도메인 이메일/위임 증빙}을 비공개로 보내주실 수 있을까요?
영업일 기준 72시간 이내에 다음 안내를 드리겠습니다.
```
```
[처리 완료]
요청하신 {대상}을(를) 제거했습니다. (커밋: {링크})
라이브 사이트에는 재배포 후 반영됩니다.
추가로 누락된 부분이 있으면 알려주세요.
```
```
[추가 증빙 요청]
권리 확인을 위해 {필요 증빙}이 더 필요합니다.
민감정보가 포함될 수 있으니 비공개 채널(Security Advisories)로 보내주세요.
```
```
[부분 수용 / 협의]
요청 중 {A}는 즉시 처리했고, {B}는 {사유: 공개 출처·식별 목적 등}로
{부분 추상화/유지}를 제안드립니다. 의견 주시면 반영하겠습니다.
```

### 7. 기록
- [CHANGELOG.md](../CHANGELOG.md) `## [Unreleased]`의 `### Removed`에 **한 줄**로 기록한다. **요청자 식별정보(실명·이메일·회사 담당자)는 적지 않는다.**
  - 예: `- {브랜드} 로고/항목 — 권리자 요청으로 제거`
- 처리 내역은 해당 이슈/advisory의 closing 코멘트로 남긴다.
